# 리팩토링 가이드

## 핵심 원칙

- **YAGNI**: 미래를 대비한 미리 만들기 금지. 지금 필요한 것만 구현.
- **KISS**: 설명이 필요한 구조는 실패한 설계. 직관적으로.
- **선 병합, 후 분리**: 한 파일 800줄 이상이거나 명확한 재사용 포인트가 보일 때만 분리.
- **Dead Code 즉시 삭제**: 미사용 import, 주석 처리된 코드, 쓰지 않는 변수.
- **불필요한 예외처리 금지**: try/except, fallback, 방어적 validation 남발 금지. 에러가 삼켜지면 디버깅이 불가능해진다. 예외처리는 시스템 경계(CSV 입력, 파일 I/O 등)에서만 최소한으로.

## 명명 규칙

- 모호한 이름 금지: `data`, `handle`, `temp` → `feature_matrix`, `train_labels`, `kernel_gram`
- 함수/변수명만으로 역할(What)과 이유(Why)가 드러나야 함
- 수식 변수는 논문/교재 표기와 맞추되 (`alpha`, `b`, `C`, `gamma`), 1글자 변수는 좁은 스코프에서만 사용

## SVM 구현 시 추가 가이드

- **수치 안정성**: 커널 행렬은 `float64` 로 유지, 지수항은 클리핑. RBF 커널은 `||x-y||²` 를 한 번에 broadcast 로 계산.
- **검증 수단 확보**: 토이 데이터셋 (선형 분리 가능한 2D 점 등) 으로 결정 경계가 기대대로 그려지는지 먼저 확인한 뒤 실제 배터리 데이터로 확장.
- **재현성**: random seed 고정. train/val 분할도 동일 seed 로 재현 가능해야 함.
- **모델 직렬화**: numpy `.npz` 또는 `pickle` 로 가중치·서포트벡터·하이퍼파라미터·라벨 인코더 일괄 저장.

## 리팩토링 프로세스

1. 현상 파악 — 중복/복잡한 조건문/성능 저하 지점 식별
2. 검증 수단 확보 — 동일 입력에 대한 학습/예측 결과가 리팩토링 전후로 동일한지 비교 가능한 환경 확보
3. 최소 단위 실행 — 함수/클래스 단위로 한 블록씩
4. 중간 검증 — 수정 직후 토이 데이터 + 실데이터로 사이드 이펙트 확인
5. 커밋 — `refactor:` 또는 `perf:` 접두사 사용

## 적용 범위

| 모듈 | 대상 |
| --- | --- |
| 데이터 로딩 | CSV 파서, 전처리(스케일링/인코딩), train/val 분할 |
| SVM 코어 | 선형 / 소프트 마진 / 커널 SVM, 다중 클래스 전략 |
| 학습 스크립트 | `train.py` (CSV → 모델 파일) |
| 예측 스크립트 | `predict.py` (모델 + CSV → 예측 CSV) |

**성공 기준**: 학습 코드가 동일 입력·동일 seed 에서 동일한 모델을 생성하고, 예측 코드가 동일 모델·동일 입력에서 동일한 출력 CSV 를 생성.
